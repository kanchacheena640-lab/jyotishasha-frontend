## How to Use This Document

This document answers exactly one question: **What sections should every page contain and in what order?**

It does not tell you how to write. That is CONTENT_RULES.md.  
It does not tell you what the engine renders. That is `lib/authority-engine/`.  
It tells you, given a content type and authority level, which blocks appear, what fields they contain, and in what sequence they are assembled.

Use the **Block Selection Matrix (Section 7)** to determine which blocks apply to your content type. Use the **Block Specifications (Section 3)** to know what to put inside each block. Use the **Page Assembly Algorithm (Section 5)** to order and validate the assembled page.

## 1. Page Architecture Overview

A Jyotishasha authority page is assembled from four groups. The groups render in this fixed top-to-bottom order:

```
PAGE
├── HEADER GROUP  ──────────────────────────── hero.headline / hero.subtext / hero.badge
│                                               (not sections; rendered by the hero component)
│
├── CONTENT GROUP  ─────────────────────────── sections[]: 1 to 7 blocks
│   Required blocks come first, then conditional, then optional.
│   Ordering within each tier follows Section 6 of this document.
│
├── CONTEXT GROUP  ─────────────────────────── sections[]: 0 to 4 blocks
│   Contextual modifiers. Always placed after all CONTENT GROUP blocks.
│
├── FAQ BLOCK  ──────────────────────────────── sections[]: always last in sections[]
│   Required on all pages. Always the final entry in sections[].
│
└── SUPPORT GROUP  ─────────────────────────── ctas[] + auto-generated elements
    CTA block: ctas[] array (separate from sections[])
    Related Topics: auto-generated by the engine from domain registry
    Related Tools: rendered from ctas[] or a tool-link widget
```

**Critical constraint:** The `sections[]` array — which includes CONTENT blocks, CONTEXT blocks, and the FAQ block — must contain a minimum of 2 entries and a maximum of 8 entries. This is enforced by `DOMAIN_TOPIC_CONSTRAINTS.sections`. The `ctas[]` array does not count toward this limit.

**Hero fields and ctas[] are authored but are not part of sections[].** They are separate top-level fields on `DomainTopic`.

## 2. Header Group — `hero` Fields

The header group is not rendered via `sections[]`. It is rendered by the hero component from the `hero` field on `DomainTopic`. The fields within `hero` are:

### 2.1 `hero.headline` / `hero.headline_hi`

**Engine field:** `DomainTopic.hero.headline`  
**Constraint:** 40–90 characters  
**Rule:** This is the H1 of the page. It must be the primary keyword phrase in natural language, not a head term alone.  
**Required:** Yes — every page  

Format pattern by content type:

| Content Type | Pattern | Example |
|---|---|---|
| Planet in House | `[Planet] in [Ordinal] House: [Core Effect]` | "Sun in 10th House: Career, Authority and Fame in Vedic Astrology" |
| Planet in Sign | `[Planet] in [Sign]: [Core Trait]` | "Mars in Scorpio: Intense Drive, Power and Hidden Strength" |
| Ascendant | `[Sign] Ascendant: [Personality Core]` | "Scorpio Ascendant: Personality, Traits and Life Path" |
| Nakshatra | `[Name] Nakshatra: [Core Theme]` | "Ashwini Nakshatra: Healing, Speed and New Beginnings" |
| Yoga | `[Yoga Name]: [What It Gives]` | "Gaj Kesari Yoga: Wealth, Wisdom and Social Status" |
| Dasha | `[Planet] Mahadasha: [Core Effect Period]` | "Saturn Mahadasha: Lessons, Delays and Lasting Gains" |
| Transit | `[Planet] Transit [Through/Over] [Target]: [Effect]` | "Jupiter Transit Over Natal Moon: Emotional Growth and Opportunity" |
| Panchang | `[Element]: [What It Governs]` | "Tithi in Panchang: Lunar Day and Its Auspicious Significance" |
| Muhurat | `[Activity] Muhurat: [What to Consider]` | "Marriage Muhurat: Auspicious Timing for Wedding Ceremonies" |
| Vastu | `[Direction/Element] Vastu: [Governing Principle]` | "North Direction Vastu: Wealth, Career and Energy Flow" |
| Numerology | `Life Path [Number]: [Core Meaning]` | "Life Path 7: Spirituality, Introspection and Deep Wisdom" |
| Compatibility | `[Sign/Number] and [Sign/Number] Compatibility: [Axis]` | "Aries and Libra Compatibility: Attraction, Tension and Balance" |
| Gemstone | `[Gem]: [Planet] Gemstone Benefits and How to Wear` | "Blue Sapphire: Saturn Gemstone Benefits and How to Wear" |
| Remedies | `[Remedy Type] for [Problem/Planet]: [Core Benefit]` | "Hanuman Chalisa for Mars: Removing Obstacles and Building Courage" |

### 2.2 `hero.subtext` / `hero.subtext_hi`

**Engine field:** `DomainTopic.hero.subtext`  
**Constraint:** 100–280 characters  
**Rule:** The introductory paragraph. One to three sentences. States what this topic is, who it matters to, and the primary insight in plain language. Must stand alone without requiring the headline for context. No bullet points. No heading markers. Written in prose.

**What must appear in `hero.subtext`:**
- The subject named explicitly (not pronoun-only)
- At least one concrete effect or significance
- The astrological or numerological system it belongs to

**What must not appear in `hero.subtext`:**
- Lists
- Predictions stated as certainties
- Generic astrology filler ("astrology is an ancient science")
- Links (this is a prose paragraph, not a navigation element)

### 2.3 `hero.badge` / `hero.badge_hi`

**Engine field:** `DomainTopic.hero.badge`  
**Constraint:** 10–40 characters. Optional.  
**Rule:** A short category label displayed as a badge on the hero card. Examples: "Financial Astrology", "Nakshatra", "Vastu Shastra". Use the domain's canonical category name. Do not use topic-specific phrases here.

### 2.4 `hero.icon`

**Engine field:** `DomainTopic.hero.icon`  
**Constraint:** Single emoji. Optional.  
**Rule:** Thematic emoji representing the broad category, not the specific topic. Consistent across all topics in the same domain. Examples: `🪐` for planetary topics, `🏠` for Vastu, `💎` for Gemstones.

## 3. Block Specifications

Each block is specified with:
- Block ID (the value to use for `PageSection.id`)
- `PageSection.layout` value
- Item count bounds
- Required and optional SectionItem fields
- Word count per `item.body`
- `badgeVariant` usage rules
- Placement constraints

### Block Field Reference for SectionItem

Every item in every section is a `SectionItem`. Available fields:

| Field | Type | Notes |
|---|---|---|
| `id` | `string` | Unique within the section. Kebab-case. |
| `label` | `string` | Short label or question. 3–80 characters. |
| `label_hi` | `string` | Hindi translation of `label`. Required when `label` is present. |
| `body` | `string \| undefined` | Main prose content. Length varies by block specification. |
| `body_hi` | `string \| undefined` | Hindi translation of `body`. Required when `body` is present. |
| `icon` | `string \| undefined` | Single emoji. Decorative only. Not required on every item. |
| `badge` | `string \| undefined` | Short text for the badge pill. 3–20 characters. |
| `badge_hi` | `string \| undefined` | Hindi translation of `badge`. Required when `badge` is present. |
| `badgeVariant` | `SectionItemBadgeVariant \| undefined` | `positive \| negative \| neutral \| warning \| info` |
| `href` | `string \| undefined` | Internal link path. Format: `/en/[domain]/[slug]`. Only on link-eligible items (see Section 9). |

### BLOCK 01 — KEY SIGNALS

**Block ID:** `key-signals`  
**Layout:** `cards` (SectionLayout.Cards)  
**Section title:** "Key Signals at a Glance" / "मुख्य संकेत"  
**Item count:** 3–5  
**Columns:** 2 default; 3 for 5 items  
**Badge variants:** positive, negative, neutral, warning — one per card to color-code signal type  
**Badge required:** Yes — every item must have `badge` and `badgeVariant`

This block is a quick-reference summary of the page's most important signals or takeaways. It appears at the top of the CONTENT GROUP when used, acting as a "what you need to know" snapshot before the detailed content.

**SectionItem fields:**

| Field | Rule |
|---|---|
| `label` | The signal or key point. 4–10 words. |
| `label_hi` | Required. |
| `body` | One sentence expanding the signal. 15–30 words. |
| `body_hi` | Required. |
| `icon` | Optional. Thematic emoji for the signal type. |
| `badge` | Short category label for the signal. E.g., "Career", "Health", "Timing". |
| `badge_hi` | Required. |
| `badgeVariant` | Required. Reflects the nature of the signal: `positive` for benefits, `negative` for challenges, `warning` for timing cautions, `neutral` for factual points, `info` for procedural notes. |
| `href` | Not used on this block. |

**Usage rule:** Use KEY SIGNALS when `authorityLevel` is `comprehensive` and the page has 6+ sections, or when the topic is complex enough that a reader needs orientation before the detail. Do not use on introductory pages — they benefit from flowing prose, not a dashboard.

### BLOCK 02 — KEY TAKEAWAYS

**Block ID:** `key-takeaways`  
**Layout:** `checklist` (SectionLayout.Checklist)  
**Section title:** "Key Takeaways" / "मुख्य बिंदु"  
**Item count:** 4–7  
**Badge variants:** Not used on this block.  
**Badge required:** No

A structured summary of the page's conclusions. Each item is a complete standalone insight, not a header. Can be placed at the top (as a preview) or at the bottom (as a synthesis). Not both.

**SectionItem fields:**

| Field | Rule |
|---|---|
| `label` | A complete insight. 10–25 words. No truncated phrases. |
| `label_hi` | Required. |
| `body` | Not used on this block. Leave undefined. |
| `icon` | Optional. `✓` or category emoji. |
| `badge` | Not used. |
| `badgeVariant` | Not used. |
| `href` | Only on items where a deeper page exists. Internal link only. |

**Usage rule:** Use KEY TAKEAWAYS when the page is long (5+ sections) and the FAQ is not sufficient to orient a skimmer. Do not use both KEY SIGNALS and KEY TAKEAWAYS on the same page — they serve the same orientation function and combining them creates redundancy.

### BLOCK 03 — DEFINITION

**Block ID:** `definition`  
**Layout:** `grid` (SectionLayout.Grid) or `list` (SectionLayout.List)  
**Section title:** "What Is [Topic]?" / "[Topic] क्या है?"  
**Item count:** 2–4 (grid); 3–6 (list)  
**Columns (grid):** 2  
**Badge variants:** `info` only, or none.

Explains what the topic is at a foundational level. Required only for topics that most readers will not already understand. For Planet in House or Planet in Sign content, readers understand the concept — skip this block. For Yoga, Nakshatra, Panchang, Muhurat, Vastu, Gemstone, or Remedies, readers may need the concept defined.

**SectionItem fields:**

| Field | Rule |
|---|---|
| `label` | The dimension being defined. E.g., "System", "Origin", "Scope", "Purpose". |
| `label_hi` | Required. |
| `body` | 25–60 words. One paragraph. No bullet points within body. |
| `body_hi` | Required. |
| `icon` | Optional. Thematic. |
| `badge` | Optional. Category label if using grid layout. |
| `badgeVariant` | `info` if badge is used, otherwise omit. |
| `href` | Not used on this block. |

**Usage rule:** When `authorityLevel` is `introductory` and the content type is Nakshatra, Yoga, Panchang, Muhurat, Vastu, Numerology, Gemstone, or Remedies — DEFINITION is required. For intermediate and comprehensive authority levels, DEFINITION should be skipped (readers at that level don't need it). For Planet in House, Planet in Sign, Ascendant, Dasha, Transit, and Compatibility — DEFINITION is optional at introductory level and skipped at higher levels.

### BLOCK 04 — MEANING

**Block ID:** `meaning`  
**Layout:** `list` (SectionLayout.List) or `cards` (SectionLayout.Cards)  
**Section title:** "[Topic] Meaning in Astrology" / "[Topic] का ज्योतिषीय अर्थ"  
**Item count:** 3–6  
**Columns (cards):** 2  
**Badge variants:** `neutral`, `info`, or none

Describes what the topic signifies — not what it predicts, but what it represents. For planetary topics: the planet's core significations within the astrological system. For house topics: the domains the house governs. For Nakshatras: the deity, animal symbol, ruling planet, nature. For Yogas: the planets and houses involved.

**SectionItem fields:**

| Field | Rule |
|---|---|
| `label` | The signification category. E.g., "Career", "Relationships", "Spirituality", "Ruling Planet", "Deity". |
| `label_hi` | Required. |
| `body` | 30–70 words. Explains what this topic means in relation to this category specifically. |
| `body_hi` | Required. |
| `icon` | Optional. Thematic emoji. |
| `badge` | Optional. Short label. |
| `badgeVariant` | `neutral` or omit. |
| `href` | Not used. |

**Absorption rule:** For comprehensive pages where DETAILED INTERPRETATION already covers meaning in depth, MEANING can be merged with DETAILED INTERPRETATION by expanding its items. Do not run both blocks if they repeat the same territory — combine them into DETAILED INTERPRETATION and remove this block.

### BLOCK 05 — DETAILED INTERPRETATION

**Block ID:** `interpretation`  
**Layout:** `accordion` (SectionLayout.Accordion) for 5+ items; `cards` (SectionLayout.Cards) for 3–4 items  
**Section title:** "[Topic]: Full Interpretation" / "[Topic]: विस्तृत विवेचन"  
**Item count:** 4–8  
**Columns (cards):** 2  
**Badge variants:** Any variant. Use to signal the nature of each interpretation dimension.

The primary interpretive content of the page. Each item covers one life domain or interpretation axis. This is the block that must contain original insight — the observation that is not available on generic astrology sites (see CONTENT_RULES.md §1.2).

**SectionItem fields:**

| Field | Rule |
|---|---|
| `label` | The life domain or axis. E.g., "Career and Ambition", "Marriage and Relationships", "Health Patterns", "Financial Temperament". |
| `label_hi` | Required. |
| `body` | 60–120 words. Explains how the topic manifests in this domain. Must contain at least one specific, non-generic claim. |
| `body_hi` | Required. |
| `icon` | Required on cards layout; optional on accordion. |
| `badge` | Optional. Use to indicate the general tendency of this dimension. |
| `badgeVariant` | Optional. `positive` if this domain tends favorable, `negative` if challenging, `neutral` if balanced. |
| `href` | Eligible. If a dedicated page exists for this domain + topic combination, link from the label. See Section 9. |

**Absorption of context blocks:** Within DETAILED INTERPRETATION items, the effect of the sign the planet occupies, and the aspects it receives, may be noted within the body text as conditional statements ("when in Aries, this intensifies; when aspected by Saturn, it delays"). This absorbs SIGN INFLUENCE and ASPECT INFLUENCE as sub-notes within items, rather than requiring separate context blocks for those axes.

### BLOCK 10 — MISCONCEPTIONS

**Block ID:** `misconceptions`  
**Layout:** `comparison` (SectionLayout.Comparison)  
**Section title:** "Common Misconceptions" / "सामान्य भ्रांतियाँ"  
**Item count:** 3–5  
**Badge variants:** `negative` for the misconception, `positive` for the corrected view. Items pair as myth/reality.  
**Badge required:** Yes — every item must signal which column it belongs to.

Corrects widely repeated inaccurate claims about this topic. Each item is one misconception or correction. Items come in pairs: one `negative`-badged item (the myth) followed immediately by one `positive`-badged item (the corrected view). Total item count must be even when using the paired pattern.

**Alternative pattern:** Each item represents one myth, with the `label` as the myth and the `body` as the correction. Use this when the comparison layout is appropriate but pairing is awkward.

**SectionItem fields (paired pattern):**

| Field | Rule |
|---|---|
| `label` | The claim. For `negative` items: the myth. For `positive` items: the accurate statement. 6–20 words. |
| `label_hi` | Required. |
| `body` | 25–50 words. Expands the claim. For myths: why it persists. For corrections: the nuanced truth. |
| `body_hi` | Required. |
| `icon` | Optional. `✗` for myth, `✓` for correction. |
| `badge` | Required. "Myth" or "Reality" (or Hindi equivalents). |
| `badge_hi` | Required. |
| `badgeVariant` | Required. `negative` for myth items, `positive` for reality items. |
| `href` | Not used. |

### 4.1 CTA Block — `ctas[]` Array

The CTA block is authored in the `ctas[]` array on `DomainTopic`, which is a separate field from `sections[]` and does not count toward the section count budget.

### 4.2 Related Topics — Auto-Generated

The Related Topics block is **not authored**. It is auto-generated by the engine from the domain registry. The engine surfaces other topic slugs from the same domain that are marked `hubPriority: 'featured'`. Authors influence this by setting `hubPriority` on each topic, not by manually specifying links.

Do not create a `sections[]` entry for related topics.

### 4.3 Related Tools — From `ctas[]`

Tool links are included in the `ctas[]` array as `secondary` CTAs. There is no separate "Related Tools" section type in the engine. If a tool is relevant, add it as a secondary CTA entry.

## 5. Page Assembly Algorithm

Follow these steps in order to assemble a page from scratch:

**Step 1 — Identify Content Type**  
From the topic's domain, category tags, and `aiMetadata.searchIntent`, determine which row to use in the Block Selection Matrix (Section 7).

**Step 2 — Apply Authority Level Filter**  
The `aiMetadata.authorityLevel` field gates certain blocks:
- `introductory`: DEFINITION may be required; DIVISIONAL CHARTS is always N; KEY SIGNALS and KEY TAKEAWAYS are not used.
- `intermediate`: Standard block selection applies. DIVISIONAL CHARTS is optional.
- `comprehensive`: DIVISIONAL CHARTS is available; KEY SIGNALS is available; full context group can be used.

**Step 3 — List Required Blocks**  
From the Block Selection Matrix, extract all `R` blocks for this content type. These are mandatory — every page of this content type must contain them.

**Step 4 — Evaluate Conditional Blocks**  
For each `C` block in the matrix, check the associated condition. If the condition is met for this specific topic, add the block to the build list. If not, skip it.

Condition definitions:
- **C¹** (DEFINITION required): `authorityLevel === 'introductory'` AND content type is Nakshatra, Yoga, Panchang, Muhurat, Vastu, Numerology, Gemstone, or Remedies.
- **C²** (DISCLAIMER required): topic makes timing predictions, health claims, or financial recommendations. For Financial Astrology domain: always. For Gemstone: always. For Muhurat: always.
- **C³** (STRENGTH ANALYSIS required): topic is Planet in House, Planet in Sign, Ascendant, or Yoga AND `authorityLevel` is intermediate or comprehensive.
- **C⁴** (HOUSE OWNERSHIP EFFECT required): topic is Planet in House AND at least one ascendant produces a meaningfully different result.
- **C⁵** (DASHA INFLUENCE required): topic is Yoga, Dasha, or Transit AND timing dimension adds insight not covered in CHALLENGES.
- **C⁶** (TRANSIT TIMING required): topic is Transit or Panchang.

**Step 5 — Add Optional Blocks**  
Review the remaining `O` blocks. Select those where the specific topic has sufficient unique content to populate them at full quality. Do not add optional blocks to hit a page length target — add them only when they carry distinct information.

**Step 6 — Budget Check**  
Count all selected blocks: CONTENT + CONTEXT + FAQ + (DISCLAIMER if required). This total must be between 2 and 8. If it exceeds 8:
1. Remove optional blocks first, lowest-information-value first.
2. Remove conditional blocks that are borderline — reevaluate if the condition truly applies.
3. Absorb narrow blocks into adjacent blocks (e.g., absorb SIGN INFLUENCE into DETAILED INTERPRETATION items).
4. Never remove required blocks. Never remove the FAQ block.

If the total is below 2, the topic is too narrow for a standalone page. Merge it with a related topic page instead.

**Step 7 — Order Blocks**  
Apply the ordering rules in Section 6. CONTENT GROUP blocks come before CONTEXT GROUP blocks. FAQ is always last. DISCLAIMER, if present, is immediately before FAQ.

**Step 8 — Generate Hero Fields**  
After all sections are drafted, write `hero.headline` and `hero.subtext`. These synthesize the page's core argument — they should be the clearest expression of what makes this topic page valuable, written with knowledge of all the sections beneath them.

**Step 9 — Generate CTA Entries**  
Write `ctas[]` targeting the reader's next action. The primary CTA should correspond to what a reader with this specific question naturally wants to do next (get a full chart reading, use a tool, read a related topic).

**Step 10 — Final Validation**  
Check:
- `sections.length` is between 2 and 8.
- `sections[sections.length - 1].layout === 'faq'` (FAQ is last).
- `ctas.length` is between 1 and 3.
- Every section item has both `label` and `label_hi`.
- Every `badge` has `badge_hi`.
- Every `body` has `body_hi`.
- No section uses an ID that duplicates another section's ID on this page.
- Word count constraints are met for each block per the specifications in Section 3.

## 6. Section Ordering Rules

### 6.1 Fixed Position Rules

These are absolute — no exceptions:
- **FAQ block:** always `sections[sections.length - 1]`. Always last.
- **DISCLAIMER block (when present):** always `sections[sections.length - 2]`. Always immediately before FAQ.
- **CONTENT GROUP blocks:** always precede all CONTEXT GROUP blocks.

### 6.2 Within the CONTENT GROUP

Apply this ordering priority for content blocks:

```
1. KEY SIGNALS (if used)
2. KEY TAKEAWAYS (if used — mutually exclusive with KEY SIGNALS)
3. DEFINITION (if required)
4. MEANING (if separate from DETAILED INTERPRETATION)
5. DETAILED INTERPRETATION
6. POSITIVE INDICATORS
7. CHALLENGES
8. INFLUENCING FACTORS
9. PRACTICAL GUIDANCE
10. MISCONCEPTIONS
```

Do not reorder to create "flow". The order above is the canonical reading sequence from most fundamental to most nuanced. Readers who skip ahead (which most do) need the fundamental blocks to appear first.

**Exception — Action-first topics:** For Remedies, Muhurat, and Practical Guidance–heavy pages (where PRACTICAL GUIDANCE is the primary value), PRACTICAL GUIDANCE may be elevated to position 4 (before DETAILED INTERPRETATION) to serve readers who come specifically for the actionable method. This is the only permitted reordering exception.

### 6.3 Within the CONTEXT GROUP

```
1. STRENGTH ANALYSIS
2. HOUSE OWNERSHIP EFFECT
3. SIGN INFLUENCE
4. ASPECT INFLUENCE
5. DASHA INFLUENCE
6. TRANSIT TIMING
7. DIVISIONAL CHARTS
```

This order reflects analytical dependency: you must understand the planet's inherent strength before understanding how the ascendant changes it; you must understand how the ascendant changes it before contextualizing aspects.

### 6.4 Mixed Layouts — Consecutive Rules

Do not place two consecutive sections with the same layout if they cover related content. This creates visual monotony and confuses readers about whether they are reading the same section or a new one. Specifically:
- Do not place two consecutive `list` sections.
- Do not place two consecutive `checklist` sections.
- Do not place two consecutive `alert` sections.
- Do not place two consecutive `cards` sections.

If two required blocks would normally use the same layout and must appear consecutively, use the alternative layout option specified in the block's specification for one of them.

## 8. Section Count Budget

The total number of entries in `sections[]` — including all CONTENT blocks, all CONTEXT blocks, DISCLAIMER (when present), and the FAQ block — must be between 2 and 8.

### Typical Section Counts by Authority Level

| Authority Level | CONTENT blocks | CONTEXT blocks | FAQ | DISCL | Total |
|---|---|---|---|---|---|
| `introductory` | 2–3 | 0–1 | 1 | 0–1 | 3–5 |
| `intermediate` | 3–4 | 1–2 | 1 | 0–1 | 5–7 |
| `comprehensive` | 4–5 | 2–3 | 1 | 0–1 | 7–8 |

### Budget Priority (when over limit)

When the initial block selection exceeds 8, remove blocks in this priority order (from lowest to highest priority to keep):

1. DIVISIONAL CHARTS (always optional; highest priority to cut)
2. SIGN INFLUENCE (can be absorbed into DETAILED INTERPRETATION)
3. ASPECT INFLUENCE (can be absorbed into INFLUENCING FACTORS)
4. KEY TAKEAWAYS (useful but not unique information)
5. MISCONCEPTIONS (valuable but deferrable)
6. PRACTICAL GUIDANCE (deferrable unless Remedies or Muhurat)
7. DEFINITION (skip at intermediate/comprehensive)
8. TRANSIT TIMING (context only; skip unless page is about timing)
9. CONTEXT GROUP blocks generally (except HOUSE OWNERSHIP for Planet topics, which is near-required)
10. **Do not remove:** MEANING, DETAILED INTERPRETATION, CHALLENGES, FAQ, DASHA INFLUENCE on Yoga pages, HOUSE OWNERSHIP on Planet in House pages

## 9. Internal Linking Placement Rules

### 9.1 Hero Links

No links in `hero.subtext`. The introduction is prose — links in prose introductions create early exit points before the reader has consumed the page value.

### 9.2 Section-Level Links — Eligible Blocks

Links (`item.href`) may only be placed on items in these blocks:

| Block | Link eligibility | Target pattern |
|---|---|---|
| KEY TAKEAWAYS | Eligible when a dedicated page expands the takeaway | `/en/[domain]/[slug]` |
| DETAILED INTERPRETATION | Eligible when a specific domain-combination page exists | `/en/[domain]/[slug]` |
| CHALLENGES | Eligible when a remedies page addresses the challenge | `/en/[domain]/[slug]` |
| PRACTICAL GUIDANCE | Eligible when a tool or deeper page exists | `/en/tools/[slug]` or `/en/[domain]/[slug]` |
| HOUSE OWNERSHIP EFFECT | Eligible when an Ascendant topic page exists for that ascendant | `/en/[domain]/[slug]` |
| SIGN INFLUENCE | Eligible when a Planet in Sign page exists | `/en/[domain]/[slug]` |
| DASHA INFLUENCE | Eligible when a Dasha topic page exists for the activating planet | `/en/[domain]/[slug]` |

### 9.3 Blocks Where Links Are Prohibited

Do not set `href` on items in: DEFINITION, MEANING, POSITIVE INDICATORS, STRENGTH ANALYSIS, ASPECT INFLUENCE, TRANSIT TIMING, DIVISIONAL CHARTS, MISCONCEPTIONS, FAQ, DISCLAIMER.

These blocks are either definitional (where linking out fragments the explanation) or evaluative (where external confirmation undermines authority).

### 9.4 Link Density Rules

- Maximum 2 internal links per section.
- Maximum 6 internal links per page across all sections.
- No two items in the same section should link to the same target.
- Every `href` must point to an existing published page. Do not create speculative links to pages that do not yet exist.
- All `href` values must be absolute internal paths starting with `/en/` or `/hi/`. Never use relative paths.

### 9.5 `linkingKeywords` Field

The `DomainTopic.linkingKeywords` field is used by the content system to identify anchor text candidates for links FROM OTHER PAGES TO THIS PAGE. It is not used within this page's own sections. Populate it with 2–5 specific phrases that other pages should use when linking to this topic.

## 10. SEO Requirements

These requirements apply at the page level. They are enforced by the `DomainTopic` schema, not by sections, but content authors must ensure sections satisfy them.

### 10.1 Keyword Placement

| Location | Requirement |
|---|---|
| `hero.headline` | Primary keyword phrase must appear in the first 60 characters |
| `hero.subtext` | Primary keyword must appear within the first sentence |
| MEANING or DETAILED INTERPRETATION | Primary keyword must appear in at least one item `label` |
| First FAQ question | Must contain the primary keyword or a close variant |
| Section titles | Must not all use the primary keyword — vary with related terms |

### 10.2 `focusKeyword` Field

Every `DomainTopic` must set `focusKeyword` and `focusKeyword_hi`. This is the single primary keyword phrase the page is optimized for. It must:
- Appear in `hero.headline`
- Appear in `metaDescription`
- Appear in the first item label of the primary content block (MEANING or DETAILED INTERPRETATION)
- Not appear more than 3 times in `hero.subtext` + the first content block's items combined

### 10.3 `metaDescription` Constraint

- 140–160 characters.
- Begins with the focus keyword or a variant.
- States the page's primary value proposition.
- Does not end mid-sentence (respect the character limit).
- Does not contain markdown formatting characters.

### 10.4 `keywords` Array

- 4–10 entries.
- First entry must be the focus keyword.
- Remaining entries are semantic variants, related terms, and long-tail phrases.
- No entries that are generic ("astrology", "Vedic", "planet") without specificity.
- Entries are ordered by search volume (highest first, best guess).

### 10.5 `title` Field

- 40–60 characters.
- Keyword-first.
- Never ends with a dash or pipe — the browser appends ` | Jyotishasha` automatically.
- Different from `hero.headline` (the `title` field is the `<title>` tag; `hero.headline` is the H1).

## 11. Schema.org Placement

The engine generates Schema.org JSON-LD from `DomainTopic.schemaSignals`. Authors do not write JSON-LD directly — they populate the signal fields that the engine reads.

### 11.1 Default Schema Type

All pages render as `Article` schema unless `schemaSignals.schemaType` is set.

### 11.2 When to Use `schemaType: 'FAQPage'`

Set `schemaSignals.schemaType = 'FAQPage'` when:
- The page has 5+ FAQ items in the FAQ block, AND
- The FAQ block contains the page's primary SEO value (i.e., the topic is primarily answer-driven — common for Panchang, Numerology, basic Compatibility pages), AND
- `aiMetadata.searchIntent === 'informational'`

FAQPage schema causes search engines to render FAQ rich results in SERP. Use it when that visibility is appropriate for the topic.

### 11.3 When to Use `schemaType: 'HowTo'`

Set `schemaSignals.schemaType = 'HowTo'` when:
- The page's primary value is procedural instruction, AND
- PRACTICAL GUIDANCE is the main content block, AND
- Content type is Remedies, Gemstone (wearing method), or Muhurat (selection process)

HowTo schema is only valid when the page literally describes how to do something step by step. Do not use it for interpretive astrology pages.

### 11.4 `schemaSignals.expertise` Field

This is the E-E-A-T expertise statement. 20–120 characters. States what expertise underlies this page's content. Examples:
- "Vedic astrology analysis of planetary house placements"
- "Traditional Jyotish interpretation of Nakshatra characteristics"
- "Vastu Shastra directional analysis using classical texts"

This field is not rendered visibly to readers. It feeds structured data and content moderation tooling.

### 11.5 Schema and Section Alignment

When `schemaType` is `FAQPage`, the FAQ block must contain at least 4 items with complete `label` (question) and `body` (answer) fields — both in English. Hindi fields (`label_hi`, `body_hi`) are not included in Schema.org output by the engine.

## 12. Mobile Readability Rules

All sections are rendered using the Authority Engine's Tailwind CSS classes. These rules govern content decisions that affect mobile readability — they are content authoring rules, not CSS rules.

### 12.1 Item Label Length

Item labels render in `text-sm` (14px) on all layouts. On mobile, long labels wrap, which breaks the visual alignment of cards and grid layouts.

Maximum label word counts by layout:

| Layout | Mobile max words per label |
|---|---|
| `cards` | 6 words |
| `grid` | 8 words |
| `table` (row header cell) | 5 words |
| `timeline` | 8 words |
| `checklist` | 20 words (checklist items are longer by design) |
| `list` | 20 words |
| `comparison` | 8 words |
| `accordion` | 10 words |
| `faq` | 12 words |
| `alert` | 10 words |

### 12.2 Item Body Length for Cards and Grid

Cards (`cards` layout) on mobile render full-width (1 column). Body text that is readable in a 2-column card (≤50 words) reads comfortably on mobile. Body text over 80 words in a `cards` section will produce cards that are very tall on mobile — switch to `accordion` layout for items with longer bodies.

**Rule:** If any item in a planned `cards` section has a body over 60 words, switch the section to `accordion` layout.

### 12.3 Table Layout on Mobile

The `table` layout wraps in a horizontally scrollable container (`overflow-x-auto`). For a table to be readable without horizontal scrolling on mobile (375px width):
- Row header cell (label column): ≤12 characters
- Body cell: wraps at any width, no constraint
- Badge cell: only present when the badge variant adds necessary information

If row headers exceed 12 characters systematically, use `list` layout with badges instead of `table`.

### 12.4 Accordion vs Cards Decision Rule

Use `accordion` when:
- Item count > 5, OR
- Any item body exceeds 60 words, OR
- Items are analytical or complex (DETAILED INTERPRETATION at comprehensive level), OR
- The content is reference material that readers scan to find specific items

Use `cards` when:
- Item count is 3–5, AND
- Item bodies are 20–50 words, AND
- Items are parallel (similar length, similar complexity), AND
- Scanning all items simultaneously adds value (POSITIVE INDICATORS, CHALLENGES)

### 12.5 FAQ Item Count vs Mobile Scroll

Each FAQ item adds approximately 80–140px of height on mobile when collapsed. A FAQ section with 8 items produces a 640–1120px tall collapsed accordion. This is acceptable — do not reduce FAQ count for mobile scroll concerns. Readers on mobile expect to scroll.

## 13. Content Version and Review Rules

### 13.1 `contentVersion` Field

Start at `1`. Increment by 1 each time a substantive content revision is made (not spelling corrections, not `label_hi` updates alone).

A substantive revision is: changing a claim, adding or removing an item, changing a `badgeVariant`, changing the section layout, or adding or removing a section.

### 13.2 `lastReviewedAt` Field

ISO 8601 date string. Format: `YYYY-MM-DD`. Set to the date of the most recent human editorial review.

**Review cadence by content type:**
- Transit pages: 90 days (transit windows change)
- Panchang pages: 180 days
- All other pages: 365 days

When `lastReviewedAt` is more than 365 days old for any page, that page must enter editorial review before being promoted in internal linking or surfaced prominently in the domain hub.

### 13.3 `status` Transition Rules

```
draft → review:     content author marks complete; at least 2 sections populated
review → published: human editor confirms CONTENT_RULES.md checklist passed
published → archived: topic is superseded, domain removed, or slug changed
archived → draft:   content can be revived and revised (new contentVersion)
```

The `_index.ts` for each domain filters to `status === 'published'` before exposing topics to the engine registry. Draft and review topics do not appear on live pages.
